# v3.9.10.11 [P1-20] Re-evaluation — Honest 3-Tier

> **TL;DR**: [P1-20] S2 throttling code is **correct and tested** (8/8
> unit tests pass, 1 RPS + 1s/2s/4s backoff + 3 retries), but **without
> an S2_API_KEY env var, the S2 free tier returns 429 on every call** —
> so S2 still contributes 0 papers in the pool. Result: v3.9.10.11 data
> is essentially identical to v3.9.10.10. The v3.9.7.3 NDCG@10=0.81
> regression is NOT reversed by [P1-20] alone. **You need an S2 API key
> for [P1-20] to actually help.**

## Root cause: S2 still returns 0 papers

The v3.9.10.11 rebuild (50 queries, 6 engines with throttled S2) hit 429
on every S2 call:

```
[S2] 429 rate-limited, backing off 1s (attempt 1/4)
[S2] 429 rate-limited, backing off 2s (attempt 2/4)
[S2] 429 rate-limited, backoff 4s (attempt 3/4)
[S2] 429 rate-limited, backoff 8s (attempt 4/4) -- exhausted, returns (429, {})
```

This is **NOT a bug in [P1-20]**. The throttle and retry are working as
designed. The issue is that the S2 free tier without an API key has a
very low rate limit (looks like ~1-2 requests per 15s), and every call
in a 50-query burst hits it.

**Verification**: S2 contribution per query is `0` in `by_engine` output
(see `test_output/_rebuild_v3_9_10_11_remaining.log` and earlier log).

## 3-Tier Honest Assessment

### ✅ Works (real, verified)

1. **[P1-20] code is correct**: 8/8 unit tests pass. Throttle maintains
   1 RPS sustained. Backoff/retry handles 429 correctly. No retry on
   4xx (saves time).
2. **S2 throttling doesn't break things**: v3.9.10.11 metrics are within
   0.05 of v3.9.10.10 (essentially identical, as expected when S2
   contributes 0 papers).
3. **Ranker stability**: Intersection NDCG@10 is 0.80 (v3.9.10.11) vs
   0.82 (v3.9.7.3), still close — ranker formula itself is fine.

### ⚠ Mixed (real but with caveat)

1. **Without S2_API_KEY, [P1-20] has no effect on the pool**. S2
   returns 0 papers regardless of throttling. The fix is necessary
   infrastructure but needs the API key to actually help.
2. **Rebuild is slower** with [P1-20]: each S2 call waits up to 15s
   (4 retries * cumulative backoff) before failing. With 50 queries
   that's ~12.5 minutes wasted on S2 retries that all fail.

### ❌ Does NOT work

1. **The v3.9.7.3 NDCG@10=0.81 regression is NOT reversed by [P1-20]
   alone**. The honest truth: **without an S2 API key, the gzip/brotli
   fix in v3.9.10.10 cannot recover the relevance quality of v3.9.7.3.**
2. **Pool coverage is still -11%** (0.8873 vs 0.9971). Users will
   silently miss ~11% of label=2 papers.

## Forward path

### Option A: User sets S2_API_KEY (1-line env var)
- Get a free S2 API key at https://www.semanticscholar.org/product/api
- Set `S2_API_KEY=<key>` in env
- Re-run v3.9.10.11 rebuild — S2 should now return full results
  without 429
- **Expected**: pool coverage back to ~0.99, NDCG@10 to ~0.85
- Effort: 0 LOC; just env var + re-run

### Option B: Use a different S2 endpoint
- S2 has different rate limits for different endpoints
- `/paper/search` may be more strict than `/paper/batch`
- Effort: ~1h to add `/paper/batch` for batch queries

### Option C: Accept the regression, document trade-off
- Keep v3.9.10.11 as-is, document that "S2 free tier without API key
  is unreliable for batch use"
- Users who want full S2 benefit should get a key
- Effort: 0 LOC; doc-only

**My recommendation: Option A**. S2 free API key is free and takes 5 min
to get. The [P1-20] infrastructure is now ready; just need the key.
