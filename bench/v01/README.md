# paper-agent-bench-v0.1

**Goal**: Measure paper-agent retrieval quality on a fixed labeled query set.
This is the **ground truth** layer that all v4 improvements (PaSa-lite, MoE
routing, reranker, LTR weights) are benchmarked against.

## Why this exists

Before v4 (PaSa-lite + MoE + reranker + LTR) is built, we need a **rigorous
baseline** of v3.8.3 to know:

1. What is current 5-engine precision/recall on real research questions?
2. Where does it fail (precision vs recall, what topics, what query shapes)?
3. After each v4 module lands, **does the gain exist or is it imagined?**

Without this bench, all "+22% recall" estimates are guesses. With it, every
ship claim can be backed by held-out NDCG/recall numbers.

## Scope (v0.1)

- **50 queries**, mix:
  - 15 from public academic search benchmarks (RealScholarQuery flavor +
    BEIR/SciDocs-style)
  - 10 variants from the 3 v3.8.3 real-corpus test topics (Topic1/2/3)
  - 25 user-provided research questions (slots reserved in `queries.json`)
- **Top-30 candidates** per query (current 5-engine + dedup output)
- **3-level labels**: 0 (not relevant), 1 (marginally related), 2 (core)
- **Metrics**: Recall@10/20, Precision@10/20, NDCG@10/20, MAP@10, SR (success rate)

## Methodology

### 1. Query set construction

Each query has:

```json
{
  "id": "q001",
  "query": "original research question (natural language)",
  "topic_bucket": "ml|econ|bio|social|cross",   // for stratified analysis
  "source": "pub|v3.8.3_variant|user",
  "difficulty_hint": "broad|technical|rare_terms|methodology",
  "expected_relevant_dois": [],  // optional: known seed papers (gold)
  "notes": "free text"
}
```

### 2. Candidate pool (snapshot)

For each query, run **current 5-engine + dedup** (no PaSa-lite, no MoE, no
reranker) and snapshot the **top-30 papers** into `system_outputs/<id>.json`:

```json
{
  "query_id": "q001",
  "query": "...",
  "generated_at": "2026-07-06T11:00:00",
  "config": "pa-search-baseline-v0.1",   // matches pa_cli version
  "results": [
    {
      "rank": 1,
      "doi": "10.1234/abc",
      "title": "...",
      "abstract": "...",
      "year": 2023,
      "venue": "QJE",
      "citation_count": 124,
      "engines_found_in": ["openalex", "semanticscholar"],
      "score": 0.87
    },
    ...
  ]
}
```

This snapshot becomes the **fixed candidate pool** — any v4 module improvement
is evaluated against this same pool, so v3 vs v4 comparison is fair.

### 3. Labeling protocol

Each paper in the candidate pool gets a 0/1/2 label:

- **0** = not relevant (off-topic, different domain, wrong methodology)
- **1** = marginally related (touches the topic but doesn't address the query)
- **2** = core relevant (directly addresses the query)

**Two-step labeling**:

1. **Mavis pre-label**: Mavis (the LLM) reads title + abstract, gives a label
   and one-sentence reasoning. Saved as `labels.json` initial draft.
2. **User spot-check**: User opens `labels.json`, fixes Mavis's wrong calls,
   adds labels for papers Mavis skipped. ~3 hours for 1500 papers.

### 4. Metrics

Computed per-query, then aggregated:

| Metric | Formula | K |
|---|---|---|
| Recall@K | \|relevant∩top-K\| / \|relevant\| | 10, 20 |
| Precision@K | \|relevant∩top-K\| / K | 10, 20 |
| NDCG@K | DCG@K / IDCG@K | 10, 20 |
| MAP@K | mean of average precision across queries | 10 |
| SR (Success Rate) | fraction of queries with ≥1 label-2 paper in top-20 | - |

Label-2 papers count as "relevant". Label-1 papers contribute to NDCG as
graded relevance (gain 1) but not to recall/precision (binary relevance).

## How to run

### Run baseline (v3.8.3)

```powershell
cd "G:\minimax - workspace\Paper agent"
# for each query, snapshot top-30 candidates
python -m bench.v0.1.snapshot --queries bench/v0.1/queries.json --out bench/v0.1/system_outputs
```

### Pre-label with Mavis (LLM judge)

```powershell
# Open each system_output, get abstract → Mavis labels 0/1/2 + reasoning
python -m bench.v0.1.prelabel --system-outputs bench/v0.1/system_outputs --out bench/v0.1/labels.json
```

### User spot-check

Open `bench/v0.1/labels.json`, manually correct Mavis's wrong labels.

### Compute baseline metrics

```powershell
python -m bench.v0.1.eval `
    --queries bench/v0.1/queries.json `
    --system-outputs bench/v0.1/system_outputs `
    --labels bench/v0.1/labels.json `
    --out bench/v0.1/reports/baseline_v0.1
```

Output:
- `baseline_v0.1.json` — raw metrics (per-query + aggregate)
- `baseline_v0.1.md` — human-readable summary with bar charts in markdown

## Roadmap integration

This bench is the **single source of truth** for v4 claim verification:

- `[P1-7] Iterative multi-round search` — measured against this bench
- `[P1-8] Light MoE routing` — measured against this bench
- `[P1-9] Cross-encoder reranker + BM25` — measured against this bench
- `[P1-10] LTR weight derivation` — trained ON this bench, validated on held-out

**Rule for v4 ships**: any claim like "+20% recall@20" must show
before/after numbers from this bench. Pure eyeballing or 3-query demos don't
count.
